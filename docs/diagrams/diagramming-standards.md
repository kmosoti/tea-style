# Diagramming Standards

## Purpose
Diagrams should clarify structure or flow. If a diagram increases ambiguity, it should be simplified or removed.

## Preferred tooling
Prefer accessible, versionable, and open approaches where practical:
- Mermaid
- PlantUML
- draw.io with committed source
- plain markdown tables for simple mappings

## Diagram types
Use:
- architecture diagrams for system structure
- workflow diagrams for process flow
- sequence diagrams for time-ordered interactions

## Conventions
- Keep labels short.
- Name files by subject, not by meeting date.
- Show boundaries clearly.
- Show direction of control and data flow.
- Avoid crossing lines where possible.

## Anti-patterns
- giant all-in-one architecture diagrams
- unexplained acronyms
- decorative icons that do not aid understanding
- diagrams that cannot be edited from source control
