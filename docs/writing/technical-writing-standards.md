# Technical Writing Standards

## Purpose
Technical writing in this repository should make design intent, tradeoffs, and review expectations easy to understand.

## Rules
- Write directly.
- Prefer short sections over long undifferentiated prose.
- Distinguish principles from implementation details.
- State tradeoffs and anti-patterns, not just ideals.
- Use examples to make doctrine concrete.
- Avoid motivational filler and vendor marketing language.

## Document structure
A strong doctrine document usually contains:
1. purpose
2. principles
3. preferred patterns
4. anti-patterns
5. review criteria
6. examples

## Writing examples
Examples should:
- stay small
- show the preferred shape
- explain why the example is good
- avoid toy cleverness

## Operational documents
Operational documents should identify:
- scope
- inputs and dependencies
- failure points
- rollback or recovery path
- review checkpoints
